> 國立台北科技大學 · 互動設計系專題

## 一、核心設計理念

這個專案的出發點不只是做一個「會回答問題的 Chatbot」，而是想讓使用者感覺是在跟一個**有個性的人**對話。

所以整個系統從一開始就分成兩個關注點：

1. **知識的準確性** — 回答關於北科的資訊要正確
2. **角色的一致性** — 白可奈說話的語氣、情緒要符合她的人物設定

這兩件事如果全部塞進同一個 Prompt，LLM 很容易顧此失彼。因此我選擇把它們**拆成兩個獨立階段**來處理。

---

## 二、為什麼用 LangGraph 做工作流？

### 問題起點

最初的想法很直覺：使用者問問題 → LLM 回答。但馬上遇到幾個問題：

- LLM 憑訓練資料回答，北科的細節資訊根本不夠
- 直接在 Prompt 裡塞大量資料，token 爆炸又難以維護
- LLM 需要決定「要不要查資料」，這個判斷本身也是一種邏輯

### 解法：FSM 工作流

所以我用 **LangGraph** 建了一個有限狀態機（FSM），把整個對話流程分成三個節點：

```
使用者輸入
    ↓
[start_node] → LLM 判斷：要用工具？還是直接回答？
    ↓                      ↓
[tool_node]          [style_node]
查向量資料庫              套用白可奈語氣
    ↓                      ↓
回到 start_node          回傳最終答案
（帶著查到的資料再判斷一次）
```

**tool_node 會循環**這件事很重要。LLM 拿到工具結果後，會再次判斷「這樣夠了嗎？」，不夠就繼續呼叫工具，夠了才進 style_node。這讓系統具備一定的自我修正能力。

### LLM 的回覆格式設計

為了讓 LangGraph 能夠解析 LLM 的意圖，我強制 LLM 只能用兩種 JSON 格式回覆：

```json
// 格式一：想用工具
{
  "reason": "需要查詢北科資訊",
  "action": "get_ntut_info",
  "action_input": "光華館在哪裡"
}

// 格式二：有答案了
{
  "action": "Final Answer",
  "answer": "光華館在..."
}
```

這樣 `is_use_tool()` 這個條件邊（conditional edge）只需要讀 `action` 欄位就能決定下一步，邏輯非常清晰。

## 三、知識庫的設計思路

### 為什麼用向量資料庫？

北科的資訊（各館介紹、系所說明、活動等）是非結構化的文字，傳統關鍵字搜尋很難處理語意上的相似性。向量資料庫可以做到「使用者問光華館，就算沒有完全一樣的詞也能找到相關文件」。

### 兩個 Collection 分開的原因

```
ntut_collection      → 校園知識（光華館、互動設計系...）
shirokana_collection → 白可奈的人物設定
```

我把它們**刻意拆開**，原因是：

- 知識庫會需要定期更新（新活動、新資訊），但角色人設不應該因此被污染
- 查詢時的需求不同：知識查詢要廣（k=5），人設查詢只需要少量精準內容（k=3）
- 日後如果要換一個新角色，只需要替換 `shirokana_collection`，知識庫完全不用動

### 資料庫重載機制

`db_setting.py` 裡的 `reload_datas_from_name()` 會先刪除舊資料再重新寫入，而不是直接 append，這樣避免重複資料累積導致檢索結果混亂。

## 四、角色風格系統

### 兩階段生成

這是整個系統設計中我最在意的部分。

**第一階段**（`start_node`）：LLM 只負責找到**正確的答案**，完全不考慮語氣。

**第二階段**（`style_node`）：把第一階段的答案交給另一個 LLM 呼叫，專門負責把語氣轉成白可奈的風格，並判斷情緒。

```python
# style_node 的輸出
{
    "answer": "從校門直走看到超商後左轉就到囉！旁邊有個大水池，需要我帶路嗎？",
    "mood": "開心"
}
```

### 為什麼情緒判斷放在這裡？

因為情緒是跟**說話內容**相關的，而不是跟**原始問題**相關的。讓 style_node 在知道最終要說什麼之後再判斷情緒，比一開始就猜情緒要準確得多。

情緒標籤（開心／難過／害羞／生氣／驕傲）會回傳給 Unity 前端，驅動角色動畫。

## 五、語音系統的設計

### 為什麼要兩套 TTS？

| 語言 | 引擎 | 原因 |
|------|------|------|
| 中文、英文 | XTTS v2（自架） | 支援音色複製，可以上傳參考音檔讓聲音更符合角色設定 |
| 日文 | VOICEVOX | 日文語音品質更自然，且支援細緻的語調控制 |

### 情緒驅動語音參數

語音不只是把文字念出來，不同情緒應該聽起來不一樣。我在 `vocal_mod.py` 建了一張對應表：

```python
MOOD_STYLE_MAPPING = {
    '開心': {'speedScale': 1.00, 'pitchScale': 0.2,  'intonationScale': 1.04},
    '難過': {'speedScale': 0.92, 'pitchScale': -0.04, 'intonationScale': 0.08},
    '害羞': {'speedScale': 0.95, 'pitchScale': 0.06,  'volumeScale': 0.7},
    '生氣': {'speedScale': 1.08, 'pitchScale': 0.03,  'volumeScale': 1.1},
    '驕傲': {'speedScale': 1.0,  'pitchScale': 0.00,  'intonationScale': 1.04},
}
```

這些參數目前只套用在 VOICEVOX（日文），XTTS 的情緒調整是透過 `speed` 參數做簡單控制。

### 翻譯流程

當前端請求的語音語言和原始回答語言不同時，系統會先呼叫 LLM 翻譯，再送去合成：

```
白可奈中文回答
    ↓
ai_translate() → Gemini 翻譯成日文／英文
（參考詞彙對照表，確保北科專有名詞翻譯正確）
    ↓
generate_vocal() → TTS 合成
```

翻譯 Prompt 特別要求輸出要**口語化**，因為書面語給 TTS 念起來會很奇怪。

## 六、錯誤處理與格式解析

### JSON 解析的挑戰

LLM 有時候會在 JSON 外面多包一層 markdown code block（` ```json ... ``` `），或者乾脆輸出純文字。`formate_funtion.py` 用了三層降級策略來處理：

```
1. 先用 LangChain 的 parse_json_markdown() 解析
    ↓ 失敗
2. 手動找 ```json 標記，切出內容再 json.loads()
    ↓ 失敗
3. 直接 json.loads() 嘗試
    ↓ 全部失敗
4. 包成 {"content": 原始字串} 回傳，讓流程繼續
```

這樣做的目的是**讓系統盡量不崩潰**，即使解析失敗也能繼續往下走。

### 工具錯誤處理

`use_tool()` 在工具執行失敗時不會拋出例外，而是回傳一個帶有 `error` 鍵的 dict，然後透過 `tool_response_error_prompt` 告訴 LLM「剛才出錯了，請重新選擇工具或直接回答」，讓系統有機會自我修正。

## 八、開發過程中的取捨

| 決策 | 選擇 | 放棄 | 原因 |
|------|------|------|------|
| LLM | Gemini 2.5 Flash（雲端） | Ollama Gemma3 12B（本地） | 本地模型回應速度較慢，且在 JSON 格式遵守上較不穩定 |
| 工作流框架 | LangGraph | 純 LangChain Chain | LangGraph 的 FSM 設計讓工具循環更直觀，狀態也更好追蹤 |
| 向量資料庫 | ChromaDB | 其他方案 | 本地部署方便，與 LangChain 整合成熟 |
| TTS | XTTS v2 + VOICEVOX | 單一方案 | 各有擅長語言，雙引擎能同時兼顧品質與靈活性 |
| 角色生成 | 兩階段（知識→風格） | 單一 Prompt | 單一 Prompt 難以同時保證資訊準確與語氣一致 |

## 九、目前已知限制

- **無對話歷史記憶**：每次請求都是獨立的，LLM 不記得上一輪說了什麼
- **XTTS 情緒調整有限**：目前只有 speed 參數，沒有像 VOICEVOX 那樣細緻的語調控制
- **工具循環上限未設**：理論上 FSM 可能無限循環，正式部署應加上最大迭代次數限制
- **日文情緒語音限定**：情緒驅動語音參數目前只套用在 VOICEVOX，中英文的 XTTS 尚未支援
